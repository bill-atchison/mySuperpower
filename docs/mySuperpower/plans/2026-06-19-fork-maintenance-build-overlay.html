Problem

Every upstream sync of the mySuperpower fork is conflict-heavy because customizations edit upstream-owned files in place. The recurring pain is the four HTML-variant skills and the namespace cascade, plus the manifest / README.md / CLAUDE.md replacements.

Current state

  • main is now a clean mirror of upstream/main (v6.0.3); future upstream pulls fast-forward.
  • mysuperpower holds the customizations (built on v5.1.0; tip commit v5.1.1) and is the new rolling branch.
  • Consumption: local install via /plugin marketplace add C:\Projects\GitHub\mySuperpower (marketplace source: "./"), namespace my-superpower, HTML-native output under docs/mySuperpower/{specs,plans,implementation-notes}.
  • scripts/apply-my-superpower-brand.ps1 already performs two idempotent transforms: the namespace cascade over skills/ + hooks/session-start, and the manifest name fields.

Conflict footprint of the customizations relative to the upstream base:

KindFilesConflict risk
Namespace cascade superpowers: → my-superpower: across skill bodies High (pervasive, but mechanical)
Manifest rebrand .claude-plugin/*.json, .codex-plugin/plugin.json, .cursor-plugin/plugin.json, gemini-extension.json, package.json Medium (mechanical)
Full replacements README.md, CLAUDE.md Medium (always ours)
Deep HTML rewrites skills/{brainstorming,writing-plans,executing-plans,subagent-driven-development}/SKILL.md High (interwoven authored content)
Additive (safe) templates/*.html, docs/mySuperpower/*, brand scripts None

Goal

Make upstream syncs near conflict-free while preserving: canonical skill names that fire the HTML behavior, HTML-native output, direct local install, and automatic inheritance of upstream improvements.

Approach — pristine source + generated dist/

Keep the git-tracked tree as pristine upstream plus small owned assets. A build step assembles a branded, HTML-native plugin into dist/, which the marketplace points at. Because dist/ ships exactly one skill per canonical name, the HTML behavior fires under the canonical trigger with no collision; because source skills stay pristine, merges from upstream are clean and base improvements flow in automatically. The output stays a standard, self-contained Claude Code plugin (with branded .codex-plugin/.cursor-plugin/.opencode manifests emitted too), assembled by a build rather than hand-edited.

1. Keep upstream files pristine in source

Restore upstream versions of the four HTML skills, systematic-debugging references, manifests, README.md, and CLAUDE.md in the working source. Stop committing in-place edits to upstream-owned files.

2. Move owned content out of upstream files

Add a branding/ directory:

  • branding/README.md, branding/CLAUDE.md — your full replacements, copied into dist/ by the build.
  • branding/manifest-overrides.json — name / displayName / author / urls applied to manifests at build time.

3. Express HTML skills as overlays core

For each of the four skills, add overlays/<skill>/:

  • overlay.yml — output dir + .html extension, title suffix, template reference, and an ordered list of mechanical transforms.
  • additions.md — the new process expressed as an appended ## mySuperpower additions section (Codex-review loop, browser-preview gate), appended at the end so upstream reordering does not break it.

The build composes each output skill:

dist/skills/<skill>/SKILL.md =
    pristine upstream body
  + mechanical transforms (namespace; docs/superpowers→docs/mySuperpower; .md→.html in output steps)
  + appended "## mySuperpower additions"

Each transform asserts its anchor exists and fails loudly if upstream removed it, surfacing the rare genuine change.

4. Build script

Add scripts/build-mysuperpower.ps1 (plus a .sh twin), reusing the transforms from apply-my-superpower-brand.ps1:

clean dist/  →  copy pristine plugin tree into dist/
   (exclude overlays/ branding/ scripts/ docs/ tests/ .github/ .git/ dist/)
  →  namespace cascade over dist/  →  apply manifest overrides
  →  copy branding/ README + CLAUDE into dist/  →  apply HTML overlays  →  emit dist/

Idempotent; run directly with pwsh ./scripts/build-mysuperpower.ps1 (no package.json edit, so that file stays pristine).

5. Wire the marketplace at the repo root

Keep .claude-plugin/marketplace.json at the repo root and set its plugin source to "./dist". A relative string source resolves from the marketplace root (the dir holding .claude-plugin/), so /plugin marketplace add C:\Projects\GitHub\mySuperpower keeps working and loads the plugin from <repo>/dist. Install copies the source dir to a cache and plugins can't reference files outside it, so the build emits a fully self-contained dist/ (its own .claude-plugin/plugin.json + skills/, hooks/, assets/, and the Codex / Cursor / opencode manifests).

  • Add /dist/ to .gitignore (build locally before install).
  • Commit dist/ instead only if you later want remote GitHub install.
  • After each rebuild: /plugin marketplace update (or quick-test claude --plugin-dir ./dist).

6. Safety net — .gitattributes merge=ours for the root marketplace file

.claude-plugin/marketplace.json is the one owned file kept at the repo root (it carries your marketplace identity and the source pointer). Mark it merge=ours and register the driver with git config merge.ours.driver true so upstream edits never conflict there. Everything else owned lives under branding//overlays/ and is assembled by the build.

7. Update sync workflow (README)

git checkout main && git fetch upstream && git merge --ff-only upstream/main && git push origin main
git checkout mysuperpower && git merge main      # pristine bodies → clean
pwsh ./scripts/build-mysuperpower.ps1             # regenerate dist/
# /plugin marketplace update   (or reinstall from dist/)

Detailed implementation steps (execute later)

Assumes you are on mysuperpower with a clean tree and main as the pristine upstream mirror. Commands are PowerShell.

Phase 0 — Snapshot & capture references

  • Tag the current branded state: git switch mysuperpower; git tag pre-overlay-mysuperpower.
  • Copy the current branded skills/ into .baseline-skills/ (keep out of git) for a later diff.
  • Compute the upstream base: $BASE = git merge-base mysuperpower main.
  • Per skill, save the legacy delta: git diff $BASE mysuperpower -- skills/<skill>/SKILL.md > overlays/<skill>/legacy.diff.

Phase 1 — Restore pristine upstream sources

git checkout main -- skills/brainstorming/SKILL.md skills/writing-plans/SKILL.md `
  skills/executing-plans/SKILL.md skills/subagent-driven-development/SKILL.md `
  skills/subagent-driven-development/implementer-prompt.md skills/systematic-debugging/SKILL.md `
  skills/writing-skills/SKILL.md skills/writing-skills/testing-skills-with-subagents.md `
  skills/using-superpowers/references/gemini-tools.md
git checkout main -- .claude-plugin/plugin.json .codex-plugin/plugin.json .cursor-plugin/plugin.json `
  gemini-extension.json package.json README.md CLAUDE.md

Inspect hooks/session-start: if it only carries the namespace cascade, restore pristine and let the build re-apply it; if it has functional edits, move those into branding/ first. Leave additive assets (templates/*.html, docs/mySuperpower/*, scripts/*) untouched.

Phase 2 — Create branding/ and overlays/

  • branding/README.md, branding/CLAUDE.md — your replacements (e.g. git show pre-overlay-mysuperpower:README.md > branding/README.md).
  • branding/manifest-overrides.json — name / displayName / description / version / author / homepage / repository / license / keywords.
  • Per skill: overlays/<skill>/overlay.yml (output dir, .html ext, title suffix, template path, ordered mechanical transforms + anchors) and overlays/<skill>/additions.md (the ## mySuperpower additions block). Derive from legacy.diff, then delete the diffs.

Phase 3 — Write the build script

scripts/build-mysuperpower.ps1 (+ .sh): clean dist/ → copy the pristine plugin tree into dist/ excluding overlays/ branding/ scripts/ docs/ tests/ .github/ .git/ dist/ → namespace cascade over dist/ → apply manifest-overrides.json → copy branding/ README + CLAUDE into dist/ → apply overlays to dist/skills/<skill>/SKILL.md (assert anchors; append additions) → print summary. Idempotent and fail-loud.

Phase 4 — Wire marketplace & ignore dist/

  • Set .claude-plugin/marketplace.json → plugins[0].source = "./dist".
  • Add /dist/ to .gitignore.
  • Add .gitattributes: .claude-plugin/marketplace.json merge=ours; run git config merge.ours.driver true.

Phase 5 — Build, validate, install

pwsh ./scripts/build-mysuperpower.ps1
git --no-pager diff --no-index .baseline-skills dist/skills   # expect only intended diffs
# in Claude Code:
/plugin marketplace add C:\Projects\GitHub\mySuperpower
/plugin install my-superpower@my-superpower                    # or: claude --plugin-dir ./dist

Functional smoke test: trigger brainstorming; confirm the canonical skill fires the HTML design-doc flow into docs/mySuperpower/specs/.

Phase 6 — Conflict test, docs, commit

git checkout main; git fetch upstream; git merge --ff-only upstream/main
git checkout mysuperpower; git merge main      # expect ~zero conflicts
pwsh ./scripts/build-mysuperpower.ps1          # rebuild dist/

Update branding/README.md with the new sync + build workflow, then commit on mysuperpower.

Validation

  • Parity: dist/ skills match prior behavior (intended differences only).
  • Conflict test: merging main (v6.0.3) into mysuperpower yields ~zero source conflicts.
  • Functional: install dist/, trigger an HTML skill, confirm the canonical name fires HTML output.

Risks & trade-offs

  • Overlay transforms can miss if upstream rewrites the exact lines they target — mitigated by fail-loud anchor assertions.
  • Adds a build step and a generated dist/ to the workflow.
  • One-time migration effort to extract overlays and restore pristine sources.

Out of scope

No changes to upstream behavior; no automation of the GitHub ruleset; the first big main → mysuperpower reconciliation is tracked separately and is independent of this restructuring.
