Live Implementation Notes

The implementation-notes document fills in as each task runs, opens itself in your browser, and self-refreshes until the branch is done.

Overview

Today the mySuperpower execution skills write the implementation-notes HTML once, at the very end of a branch — after the final whole-branch review. You never see it form, and nothing shows progress while the run is underway.

This change makes the document a live artifact: it is created at the start of execution with one row per planned task, opened once in your default browser, and rewritten on disk as each task moves pending → running → done. The open tab refreshes itself every few seconds, so you watch the branch build in real time. A final pass at the end writes the outcome and strips the auto-refresh, leaving a clean static record. The behavior applies to both subagent-driven-development and executing-plans.

Chosen Approach

The document drives its own refresh — no server, no watcher, no new dependency. The controller (the inline orchestrating agent) rewrites the whole file from the task state it already tracks in its progress ledger; the browser reads that file over file:// and an embedded script reloads the page on an interval. This keeps the zero-dependency promise and works fully offline.

Two alternatives were set aside: a local web server with websocket push (adds a dependency and a long-lived process), and opening the doc only at the end (doesn't meet the "while tasks run" goal). A plain <meta http-equiv="refresh"> remains the fallback if inline JS is ever undesirable, at the cost of losing scroll position.

Lifecycle

The controller hooks the document writes into bookkeeping it already performs, so no new orchestration phase is introduced:

  1. At execution start — create the doc from the template with one row per planned task (all pending), then open it once in the default browser.
  2. Before dispatching a task's implementer — rewrite the doc marking that task running.
  3. When that task's review comes back clean — in the same step that appends the ledger line, rewrite marking it done with its commit range, one-line test summary, and review verdict.
  4. If a task is BLOCKED — mark it blocked with the reason; the doc reflects the stall instead of going silent.
  5. At the end — after the final whole-branch review, one final rewrite adds the outcome and any minor-findings roll-up, removes the auto-refresh block, and commits.

The self-refresh block

A small inline script, delimited so the finalize rewrite can drop it cleanly:

<!-- AUTO-REFRESH:START (removed on finalize) -->
<script>
  // restore scroll, then reload on an interval over file://
  const y = sessionStorage.getItem('implScroll');
  if (y) window.scrollTo(0, +y);
  addEventListener('beforeunload', () =>
    sessionStorage.setItem('implScroll', String(scrollY)));
  setTimeout(() => location.reload(), 3000);
</script>
<!-- AUTO-REFRESH:END -->

3-second interval; scroll position preserved across reloads via sessionStorage; the running task is anchored so the view stays where the action is. The finalized document contains none of this — it is a plain static file.

Data flow

The controller never reads the document back into its context. It holds the task list and per-task results in the progress ledger it already maintains, and regenerates the whole HTML from that state on each update — a single write, template plus current rows. The browser is a passive viewer: it reloads the file from disk on its timer. This one-way flow (ledger → rewrite file → browser reload) means the doc can always be rebuilt from the ledger after a compaction or resume.

Git commit cadence

Decided

Disk-only during the run, one commit at the end. The file is rewritten on disk every task (so the browser updates live), but git receives only the single finalized doc. History stays clean; the transient running/pending states never land in git.

Considered and rejected: committing after every task (captures progress in history, but adds N extra commits per branch and noise to the log).

Error handling & edge cases

  • Browser won't open (headless / no default handler) — non-fatal: the controller prints the file path and proceeds. Execution never blocks on the viewer.
  • Compaction / resume — the doc is regenerated from the progress ledger, so it survives a lost context exactly like the ledger does. No state lives only in the HTML.
  • Cross-platform open — the skill speaks in actions ("open in the default browser") with a per-OS note: Windows start "" <file>, macOS open <file>, Linux xdg-open <file>. Opened once, at start; never reopened (the tab self-refreshes).
  • Finalize is mandatory — even on a blocked/aborted branch the finalize pass runs so no delivered doc is left auto-refreshing against a file that has stopped changing.

Verification

These are behavior-shaping skill edits, so verification is by rendering and walkthrough rather than a unit suite:

  • Render the updated template from file:// offline and confirm it shows correctly with each status state and with the auto-refresh active.
  • Hand-simulate the lifecycle (pending → running → done across a few rows) and confirm the open tab reflects each rewrite and holds scroll position.
  • Confirm the finalize output contains no AUTO-REFRESH block and renders as a clean static document.
  • Confirm no external requests fire (fully offline) and the masthead/palette are unchanged from the house template.
