Overview

The implementation-notes HTML template (implementation-notes-template.html, identical copies under skills/subagent-driven-development/templates/ and skills/executing-plans/templates/) is already built as a live document: it carries a <meta http-equiv="refresh">, a Task Status table (task rows with TDD subtask rows and done / in progress / pending pills), and a Decisions & Deviations card stream, and its header documents a finalize step that removes the refresh.

But both overlays today tell the agent to create that document once, after the work is done — so the live capability is never exercised; you get a static file at the end. This change wires the overlays to drive the document live — create and open it at the start, update it as the run progresses, finalize it at the end — and adds the missing step of opening it in the browser. The template itself is reused as-is.

Chosen Approach

Reuse the existing template unchanged and rewrite the two overlay instruction files so the controlling agent treats the notes document as a live artifact. No new mechanism is invented: refresh is the template's existing <meta http-equiv="refresh">, and the doc's structure (task/subtask table, decision cards, finalize-removes-refresh) is what the template already provides.

One deliberate simplification: all writes to the document stay in the controller (subagent-driven-development) or the agent (executing-plans). Implementer subagents remain report-only — they never write the shared HTML. This avoids concurrent writers and keeps dispatch prompts clean; the controller already holds the per-task results it needs (from the implementer's report and the progress ledger) to update the rows and append cards itself.

Lifecycle (subagent-driven-development)

The controller hooks the document writes into bookkeeping it already performs:

  1. At execution start — render the template into docs/mySuperpower/implementation-notes/YYYY-MM-DD-<feature>.html with one task row (plus its TDD subtask rows) per plan task, all pending, then open it once in the default browser.
  2. Before dispatching a task's implementer — flip that task's row to in progress.
  3. When that task's review comes back clean — in the same step that appends the ledger line, flip the task row (and its subtask rows) to done · <sha> from the implementer's report, and append a Decisions card for any substantive off-spec decision, deviation, or tradeoff the report surfaced.
  4. If a task is BLOCKED — leave its row in progress and append a card describing the blocker, so the doc reflects the stall instead of going silent.
  5. At the end — after the final whole-branch review, flip the status pill to Complete, add any minor-findings roll-up, remove the <meta refresh>, and commit once.

Terminal states: done (review clean — accepted minor findings are recorded on the row/cards but do not block it), blocked (an unresolved blocker), and aborted (branch abandoned). A task with must-fix (Critical/Important) findings stays in progress through the fix and re-review loop — that is not a terminal state.

executing-plans specifics

executing-plans runs in batches with human checkpoints and often in a separate session. There is no implementer subagent, so the inline agent flips rows and appends cards itself, live:

  • At most one task is in progress at a time — tasks are worked sequentially; the doc never shows two in-progress task rows.
  • Rows flip to done as each task completes within a batch, not deferred to the checkpoint — the checkpoint is where you review, not where the doc catches up.
  • A resumed session rebuilds the table from the ledger — every task the ledger marks complete renders done, the first unfinished task becomes the next in progress, and the rest pending. The doc is reconstructed from ledger state, never appended to blindly, so a fresh session needs no prior in-memory state.

Refresh & write mechanics

  • Refresh — the template's existing <meta http-equiv="refresh" content="5"> reloads the open tab; the finalize step removes it so the delivered doc is a clean static file.
  • Atomic write — each rewrite is written to a temporary file and renamed into place, so a 5-second refresh that lands mid-write never sees a truncated document. (Cheap robustness; applies regardless of refresh mechanism.)
  • One-way flow — the controller regenerates the whole document from the progress ledger plus the latest report; it never reads the HTML back. This means the doc can always be rebuilt from the ledger after a compaction or resume.

Browser open & error handling

  • Open once, at start — the skill speaks in actions ("open in the default browser") with a per-OS note: Windows (PowerShell) Start-Process <file> or cmd /c start "" "<file>", macOS open <file>, Linux xdg-open <file>. Never reopened (the tab self-refreshes).
  • Browser won't open (headless / no default handler) — non-fatal: the agent prints the file path and proceeds. Execution never blocks on the viewer.
  • Finalize, with a crash safety net — on every normal exit (including a blocked/aborted branch) finalize removes the refresh. Because a hard crash or kill can skip it, execution start also runs a repair pass: if the existing notes doc still carries a <meta refresh> from a dead session, strip it before creating and opening the new one.
  • Compaction / resume — the doc is regenerated from the progress ledger, so it survives a lost context exactly like the ledger does. No state lives only in the HTML.

Git commit cadence

Decided

Disk-only during the run, one commit at the end. The file is rewritten on disk every task (so the browser updates live), but git receives only the single finalized doc. History stays clean; the transient in-progress/pending states never land in git.

Considered and rejected: committing after every task (captures progress in history, but adds N extra commits per branch and noise to the log).

Verification

These are behavior-shaping overlay edits, so verification is by rendering and walkthrough rather than a unit suite:

  • Run build-mysuperpower and confirm each rewritten overlay's text is appended to the corresponding dist SKILL.md.
  • Render the template from file:// offline and confirm it shows correctly with each pill state and with the <meta refresh> active.
  • Hand-simulate the lifecycle (pending → in progress → done across a few rows, plus a decision card) and confirm the open tab reflects each rewrite.
  • Confirm the finalize output contains no <meta refresh> and renders as a clean static document.
  • Confirm no external requests fire (fully offline) and the masthead/palette are unchanged from the house template.
