Overview

Make this repository mySuperpower: a single, locally-installed Claude Code plugin whose brainstorm → plan → execute → subagent-dev workflow produces self-contained HTML documents instead of markdown — while remaining a clean fork of obra/superpowers that can absorb upstream improvements.

The repo currently mirrors upstream Superpowers v5.1.0 exactly, plus four untracked HTML skill variants (brainstorming-html, writing-plans-html, executing-plans-html, subagent-driven-development-html) created in an earlier session. Those four are the heart of "your version" — they capture the same methodology but emit distinctive, offline HTML specs/plans/notes rather than markdown.

This design replaces the four upstream markdown skills with your HTML versions, renames the whole plugin to my-superpower (branded mySuperpower), sets it up for local installation, and establishes a low-friction workflow for pulling future changes from upstream.

Chosen Approach

Five decisions were settled during brainstorming. They are intentionally biased toward two goals at once: a genuinely personal plugin, and the smallest practical pain when merging from upstream.

Why takeover over an overlay

The HTML content occupies the canonical skill names (brainstorming, writing-plans, executing-plans, subagent-driven-development) rather than living beside the originals. There is exactly one skill per name, so there is no trigger ambiguity and the existing bootstrap + chain work unchanged. Crucially, when upstream improves one of these skills, the merge conflict lands on the file you forked — resolving it is the act of porting their improvement into your HTML version, instead of having it silently bypass you.

Why one plugin, fully renamed

In Claude Code the plugin's name field is also the skill namespace, and a namespace is per-plugin, not per-skill. Plugin names must be kebab-case, so the technical name is my-superpower, with displayName: "mySuperpower" carrying the brand in the /plugin UI (requires Claude Code v2.1.143+). To have every skill read as my-superpower: we rename the single plugin and cascade-update the internal superpowers: references. The cascade is bounded (~10 skill files plus the hooks bootstrap and the manifests) and automated (see Components → Rebrand script), so it stays a one-command operation on every future merge.

Architecture

The repo stays a single plugin with its existing layout — only its identity and the four workflow skills change. Nothing moves directories, so upstream's structure still aligns for clean merges of everything we do not touch.

Identity vs. mechanics

  • Plugin name / namespace: superpowers → my-superpower (kebab-case; drives the install id and the my-superpower: skill prefix), with displayName: "mySuperpower" as the UI brand.
  • Skill set: the 4 workflow skills carry your HTML behavior; the ~10 others (TDD, debugging, writing-skills, code-review, git-worktrees, finishing, dispatching, using-superpowers, verification, receiving-review) stay byte-for-byte upstream.
  • Hook: the SessionStart command path is ${CLAUDE_PLUGIN_ROOT}-based and unchanged — but the injected bootstrap text references superpowers:using-superpowers, so hooks/session-start is included in the namespace cascade.

Components

1 — Skill takeover (the 4 workflow skills)

For each skill, the -html folder's content replaces the canonical skill in place, then the -html folder is removed.

Per-skill reconciliation
SkillActionNotes
brainstormingReplace SKILL.md + visual-companion.md with HTML versions; add templates/; keep scripts/Keep spec-document-reviewer-prompt.md as a Codex fallback
writing-plansReplace SKILL.md with HTML version; add templates/Keep plan-document-reviewer-prompt.md as a Codex fallback
executing-plansReplace SKILL.md with HTML version; add templates/Clean 1:1
subagent-driven-developmentReplace SKILL.md + 3 reviewer/implementer prompts with HTML versions; add templates/HTML is a clean superset

Then two cleanup passes inside the now-canonical skills:

  • Frontmatter: name: brainstorming-html → name: brainstorming (and the other three). The description: fields already use canonical trigger phrasing, so auto-triggering is unchanged.
  • Cross-references (all files, not just SKILL.md): sweep every file in each skill directory — SKILL.md, templates/*.html, and the reviewer/implementer prompt files — rewriting every -html skill-name reference to its canonical name (e.g. "invoke writing-plans-html" → "invoke writing-plans"). The templates do reference -html names (the plan template points at subagent-driven-development-html), so a SKILL.md-only pass would leave dangling references.

2 — Rename + namespace cascade

  • name: "my-superpower" + displayName: "mySuperpower" in .claude-plugin/plugin.json (and name in .codex-plugin/plugin.json, .cursor-plugin/plugin.json, gemini-extension.json, package.json); human fields (description/author/homepage/repository) branded to you and your fork. The marketplace name also goes kebab-case (e.g. my-superpower).
  • Rewrite every superpowers:<skill> → my-superpower:<skill> across all skill files and hooks/session-start. Bare-name references ("the writing-plans skill") need no change. Preserve non-namespace occurrences: the legacy ~/.config/superpowers migration path and github.com/obra/superpowers URLs are not rewritten.
  • README.md rewritten for mySuperpower, crediting obra/superpowers as upstream and documenting the HTML workflow + the update procedure.

3 — Idempotent rebrand script

Add scripts/apply-my-superpower-brand.{sh,ps1} — a targeted, repeatable rewrite that operates only on (a) the superpowers:<skill> namespace pattern across skills + hooks/session-start, and (b) the manifest name/displayName fields. It deliberately does not do a blanket "superpowers" → "my-superpower" replace, so it never corrupts URLs or the legacy config path. It is idempotent (running it twice produces no diff) — run once during conversion and again after every upstream merge, turning the cascade from a hand-merge chore into one command.

4 — Local plugin install

Point .claude-plugin/marketplace.json at this repo as a local marketplace (marketplace name: "my-superpower", plugin name: "my-superpower", source: "./"). Install flow:

/plugin marketplace add C:\Projects\GitHub\mySuperpower
/plugin install my-superpower@my-superpower

5 — Docs path & loose ends

  • Docs output: the HTML skills write to docs/mySuperpower/{specs,plans}/ (now created). Upstream's docs/superpowers/ is left untouched.
  • Tests: upstream's tests/ are left out of scope and marked known-divergent — not ported.
  • Project CLAUDE.md: replaced with a short mySuperpower note (what this fork is, the HTML workflow, the upstream-merge steps).

Data Flow

The authoring workflow (unchanged shape, HTML output)

brainstorming → using-git-worktrees → writing-plans → subagent-driven-development / executing-plans → test-driven-development → requesting-code-review → finishing-a-development-branch

Bold stages are now your HTML skills (namespaced my-superpower:), emitting HTML to docs/mySuperpower/. The non-bold stages remain upstream skills, referenced by bare or my-superpower: names.

The update-from-upstream flow

git fetch upstream → git merge upstream/main → resolve all content conflicts (clean index) → run apply-my-superpower-brand → verify idempotent → commit

The remote is added once: git remote add upstream https://github.com/obra/superpowers.git. Critically, the rebrand script runs after all conflicts are resolved and no unmerged paths remain — never against a tree containing conflict markers (which it would rewrite on both sides). Once the index is clean, the script re-applies the namespace cascade and manifest names automatically, so the only manual work is the occasional content conflict on the four skills you deliberately forked.

Error Handling & Resilience

Codex review fallback

The HTML brainstorming and writing-plans skills use Codex (codex:rescue) as the primary document reviewer. Because Codex can be unavailable or unresponsive, the kept reviewer-prompt files become an explicit fallback:

Codex is primary; if Codex is unavailable or does not respond, dispatch a general-purpose subagent using the prompt file to perform the equivalent review. The ≤3-iteration cap applies either way.

Merge resilience

  • The rebrand script is idempotent — safe to run any number of times — but runs only on a clean index (all conflicts resolved, no unmerged paths), never on a tree with conflict markers.
  • The repo layout never diverges from upstream's, so only intentional changes ever conflict.
  • Conflicts are concentrated and meaningful: the four forked skills (content) and any namespace lines upstream happens to edit (auto-handled by the script).

Verification

The conversion is considered done only when all of the following hold:

  • All four workflow skills load with correct canonical name: frontmatter and no leftover -html cross-references in any file (SKILL.md, templates, prompts).
  • A repo-wide grep finds zero dangling superpowers: namespace references (all are my-superpower:), excluding the intentionally-preserved obra/superpowers URLs and legacy config path.
  • The rebrand script is idempotent — a second run yields no diff.
  • The plugin installs locally as my-superpower@my-superpower, shows as mySuperpower in the /plugin picker, and a clean session on "let's build X" auto-triggers my-superpower:brainstorming before any code.
  • The HTML skills write specs/plans to docs/mySuperpower/.
  • Per project workflow: /simplify is run over the changed files before the final commit.
