Decisions & Deviations

Pre-Impl · Context

Rescoped to reuse the already-live template

While writing the plan we found implementation-notes-template.html was already built for live updates (a <meta refresh>, a task/subtask status table, and a Decisions & Deviations card stream), but both overlays only invoked it after the work finished — so the capability was never exercised.

  • Deviation from the first spec: the original draft proposed a from-scratch inline-JS refresh and simpler rows. We discarded that and rescoped to "reuse the template unchanged; fix the overlays." Smaller change, keeps the richer features.
  • Mechanism: the template's existing <meta refresh content="5"> drives the live view; finalize removes it.
Decision · Off-Spec Simplification

Controller/agent owns every write

All writes to the live document stay in the controller (SDD) or the agent (executing-plans); implementer subagents remain report-only. This avoids concurrent writers to the shared HTML and keeps dispatch prompts clean — a deliberate simplification of the template's original "implementer subagent flips its own subtask rows" protocol. Task 3 (fdef9ff) updated the template's WHO-UPDATES-WHAT comment to match.

Process · Worktree Base

Worktree based on mysuperpower, not main

The native worktree default branches from origin/main, but this fork keeps main as a pristine upstream mirror — overlays/ exists only on mysuperpower. The worktree was created from mysuperpower HEAD so the edited overlays were present; branching from main would have yielded an empty overlays/.
